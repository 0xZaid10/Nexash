# ZK Circuits Overview

Nexash uses two zero-knowledge circuits written in Noir, compiled with `nargo`, and proved using Barretenberg's UltraHonk proving system. This page explains the overall circuit design philosophy and how the two circuits relate to each other.

---

## Why Two Circuits?

A single monolithic circuit could theoretically prove both identity compliance and policy compliance in one proof. Nexash uses two separate circuits for several reasons:

**Separation of concerns** — The identity proof is generated once per user per treasury and can be reused across multiple payments (within the expiry window). The policy proof is generated fresh for each payment with the specific amount and payment request ID. Keeping them separate allows the identity proof to be cached.

**Different parties** — In the institution flow, the identity proof is generated using the *recipient's* NexaID attestation data. The policy proof is generated by the *institution's* operator. Separating them means each party proves only what they know.

**Independent upgradability** — If the identity circuit needs to be updated (e.g., to support a new attestation format), only the `IdentityVerifier` needs to be redeployed. The policy circuit and its verifier are unaffected.

**Parallel generation (future)** — With Web Workers, both proofs could be generated simultaneously. Currently they run sequentially for simplicity.

---

## Circuit Language: Noir

Noir is a domain-specific language for writing ZK circuits, developed by Aztec Labs. Key properties relevant to Nexash:

**Field-agnostic** — Noir circuits are written without specifying the underlying elliptic curve. The curve is chosen at compile time (BN254 for Barretenberg/EVM).

**Rust-like syntax** — Noir uses familiar control flow (if/for/assert), making circuits readable to developers without cryptography backgrounds.

**Black-box functions** — Expensive operations like SHA-256, Keccak, secp256k1 signature verification, and Pedersen hash are provided as black-box functions — optimized implementations that Barretenberg handles efficiently.

**Assertion-based proving** — ZK circuits in Noir are collections of `assert` statements. If all assertions pass during witness generation, a valid proof can be generated. If any assertion fails, proof generation fails.

---

## Proving System: UltraHonk

UltraHonk is a PLONK-based proving system with several advantages over vanilla PLONK:

**No circuit-specific trusted setup** — UltraHonk uses a universal trusted setup (SRS) that works for any circuit up to a certain size. This eliminates the need for per-circuit ceremonies.

**Efficient for complex circuits** — UltraHonk handles lookup tables and custom gates efficiently, which is important for hash functions like Pedersen used extensively in Nexash.

**EVM-compatible proofs** — Barretenberg generates Solidity verifier contracts for UltraHonk proofs that work on any EVM chain, including HashKey Chain.

**In-browser proving** — Barretenberg provides a WebAssembly build (`@aztec/bb.js`) that runs in modern browsers without any server dependency.

## Circuit Compilation Pipeline

```
main.nr (Noir source)
    │
    │ nargo compile
    ▼
identity_compliance.json (compiled circuit)
    │ Contains: bytecode, abi, hash
    │
    ├── Deployed to: public/circuits/ (frontend)
    │
    └── export_verifiers.mjs (Barretenberg)
            │
            │ Generates VK + Solidity verifier
            ▼
        IdentityVerifier.sol
            │
            │ forge deploy
            ▼
        0x4089449207d346cDe... (HashKey Chain)
```

### Compilation

```bash
cd circuits/identity_compliance
nargo compile
# Output: target/identity_compliance.json (bytecode + ABI)
```

### VK Generation and Verifier Export

```javascript
// circuits/prover/export_verifiers.mjs
import { UltraHonkBackend } from '@aztec/bb.js'

const circuit = JSON.parse(readFileSync('./identity_compliance.json'))
const backend = new UltraHonkBackend(circuit.bytecode)

// Generate verification key
const vk = await backend.getVerificationKey()
console.log(`VK: ${vk.length} bytes`)

// Export Solidity verifier
const verifierSol = await backend.exportSolidityVerifier()
writeFileSync('./IdentityVerifier.sol', verifierSol)
```

### Deployment

```bash
forge create src/IdentityVerifier.sol:HonkVerifier \
  --rpc-url https://testnet.hsk.xyz \
  --private-key $PRIVATE_KEY
```

## Circuit Sizes

| Circuit | Constraint count (approx.) | VK size | Proof size |
|---|---|---|---|
| identity_compliance | ~65,000 | 1,888 bytes | ~2,048 bytes |
| transaction_policy | ~32,000 | 1,888 bytes | ~2,048 bytes |

The constraint count is the number of arithmetic constraints in the circuit. More constraints = more complex circuit = longer proof generation time. The Pedersen hash and Merkle tree operations dominate the identity circuit's constraint count.

## The BN254 Field

Both circuits operate over the BN254 scalar field. All inputs, outputs, and intermediate values are elements of this field — integers modulo:

```
p = 21888242871839275222246405745257275088548364400416034343698204186575808495617
```

This is approximately 2^254, meaning 256-bit Ethereum values (transaction hashes, addresses as field elements) can exceed the field modulus and must be reduced before use in the circuit.

All address-to-field conversions in Nexash apply `% p` to ensure valid field elements.

## Testing

Both circuits include test cases verified with `nargo test`:

```bash
cd circuits/identity_compliance
nargo test --show-output

[identity_compliance] Testing test_valid_proof_with_nexaid_attestation ... ok
[identity_compliance] Testing test_kyc_level_too_low ... ok
[identity_compliance] Testing test_zero_report_tx_hash_fails ... ok
[identity_compliance] 3 tests passed
```

Test cases use small field elements and hardcoded values to verify circuit logic without requiring real NexaID attestations.
