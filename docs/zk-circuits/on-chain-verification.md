# On-Chain Verification

Both UltraHonk proofs generated by Nexash are verified on HashKey Chain by Solidity verifier contracts exported from Barretenberg. This page explains how on-chain verification works, what the verifier contracts do, and how public inputs are structured.

---

## How UltraHonk Verification Works

UltraHonk is a SNARK proving system. The verifier does not rerun the computation — it checks a compact cryptographic proof that the prover executed the circuit correctly with the claimed public inputs.

The Solidity verifier performs:

1. **Transcript hashing** — Builds a Fiat-Shamir challenge from the proof and public inputs
2. **KZG polynomial commitments** — Checks that the prover committed to the correct polynomial evaluations
3. **Pairing check** — Verifies the polynomial evaluation claims using BN254 elliptic curve pairings
4. **Returns bool** — `true` if the proof is valid for the claimed public inputs, `false` otherwise

The verifier is deterministic and trustless — anyone can call it with any proof and get the same result. There is no trusted setup involved (UltraHonk uses a universal trusted setup, not circuit-specific).

---

## Verifier Contract Generation

The verifier contracts are generated by Barretenberg's `nargo` toolchain:

```bash
# Compile the circuit
cd circuits/identity_compliance
nargo compile

# Export the Solidity verifier
cd circuits/prover
node export_verifiers.mjs
# Outputs: IdentityVerifier.sol, PolicyVerifier.sol
```

`export_verifiers.mjs` uses `@aztec/bb.js` to:
1. Load the compiled circuit bytecode
2. Generate a verification key (VK) — a compact representation of the circuit's structure
3. Export the VK and verification logic as a Solidity contract

The VK is embedded directly in the Solidity contract. If the circuit changes, the VK changes, and a new verifier must be deployed.

## Public Input Layout

Public inputs are passed as a `bytes32[]` array to the verifier. The order matters — the contract reads specific indices.

### Identity Proof Public Inputs (7 inputs)

| Index | Name | Description |
|---|---|---|
| 0 | `min_kyc_level` | Minimum required KYC tier |
| 1 | `allowed_jurisdictions_root` | Merkle root of allowed countries |
| 2 | `nullifier` | Pedersen(nullifier_secret, treasury_address) |
| 3 | `treasury_address` | Registry or treasury this proof is for |
| 4 | `proof_timestamp` | Unix timestamp of proof generation |
| 5 | `expiry_window` | Validity duration in seconds |
| 6 | `report_tx_hash_public` | NexaID attestation tx hash (verifiable on chain) |

### Policy Proof Public Inputs (7 inputs)

| Index | Name | Description |
|---|---|---|
| 0 | `spending_limit` | Per-transaction maximum |
| 1 | `daily_spend_limit` | Daily maximum |
| 2 | `multisig_threshold` | Required approvals |
| 3 | `min_operator_role` | Minimum role |
| 4 | `allowed_jurisdictions_root` | Must match identity proof |
| 5 | `policy_hash` | Pedersen hash of full policy |
| 6 | `payment_req_id` | Unique payment identifier |

## ZKTreasury Verification Logic

```solidity
function executePayment(
    address token,
    address recipient,
    uint256 amount,
    bytes32 paymentReqId,
    bytes calldata identityProof,
    bytes32[] calldata identityPubInputs,
    bytes calldata policyProof,
    bytes32[] calldata policyPubInputs
) external onlyOperator nonReentrant whenNotPaused {
    
    // 1. Input count validation
    require(identityPubInputs.length == 7, "ZKTreasury: wrong identity input count");
    require(policyPubInputs.length   == 7, "ZKTreasury: wrong policy input count");

    // 2. Extract nullifier and check replay
    bytes32 nullifier = identityPubInputs[2];
    require(!usedNullifiers[nullifier], "ZKTreasury: nullifier already used");
    
    // 3. Check payment request not duplicate
    require(!usedPaymentRequests[paymentReqId], "ZKTreasury: payment request already used");

    // 4. Check proof timestamps
    uint256 idTimestamp  = uint256(identityPubInputs[4]);
    uint256 polTimestamp = uint256(policyPubInputs[6]);
    require(block.timestamp <= idTimestamp  + PROOF_EXPIRY, "ZKTreasury: identity proof expired");
    require(block.timestamp <= polTimestamp + PROOF_EXPIRY, "ZKTreasury: policy proof expired");

    // 5. Check policy proof is bound to current policy
    bytes32 currentPolicyHash = policyEngine.getPolicyHash(address(this));
    require(
        policyPubInputs[5] == currentPolicyHash,
        "ZKTreasury: policy proof bound to wrong policy"
    );

    // 6. Verify ZK proofs
    bool idValid  = identityVerifier.verify(identityProof, identityPubInputs);
    bool polValid = policyVerifier.verify(policyProof, policyPubInputs);
    require(idValid,  "ZKTreasury: invalid identity proof");
    require(polValid, "ZKTreasury: invalid policy proof");

    // 7. Check balance
    require(
        IERC20(token).balanceOf(address(this)) >= amount,
        "ZKTreasury: insufficient balance"
    );

    // 8. Record usage
    usedNullifiers[nullifier]          = true;
    usedPaymentRequests[paymentReqId]  = true;

    // 9. Execute transfer
    IERC20(token).transfer(recipient, amount);
    emit PaymentExecuted(paymentReqId, recipient, amount);
}
```

## Gas Cost Breakdown

A full payment transaction (`executePayment`) costs approximately 750,000 gas:

| Step | Gas |
|---|---|
| Input validation and checks | ~20,000 |
| `identityVerifier.verify()` | ~300,000 |
| `policyVerifier.verify()` | ~300,000 |
| Storage writes (nullifier, payReqId) | ~40,000 |
| `IERC20.transfer()` | ~30,000 |
| Events and overhead | ~10,000 |
| **Total** | **~700,000** |

At HashKey Chain Testnet gas prices (~1 gwei), this costs approximately 0.0007 HSK per payment — less than $0.01 at current HSK testnet prices.

## Local Verification

Before submitting a proof on-chain, Nexash performs local verification in the browser:

```typescript
const verified = await backend.verifyProof(
  { proof, publicInputs },
  { verifierTarget: 'evm' }
)

if (!verified) throw new Error('Proof failed local verification')
```

This catches invalid proofs before paying gas for an on-chain transaction that would fail. The local verifier uses the same verification logic as the Solidity contract — if local verification passes, on-chain verification will pass (assuming the public inputs match what the contract expects).
