# Trust Model

Understanding what Nexash trusts, what it verifies, and what it does not need to trust at all.

---

## What Nexash Does Not Trust

### No Trust in NexaID's Attestor Key

The NexaID attestor (`0x154ce2f65d15f81de926dfc91e6facd706b77441`) signs attestations with a key stored inside a Phala TEE. Nexash does not attempt to verify this signature. Instead, Nexash trusts the `reportTxHash` — a confirmed transaction on HashKey Chain. If the transaction exists and is confirmed, the attestation happened. The chain itself is the trust anchor, not a cryptographic keypair.

This design choice makes Nexash resilient to key rotation, key compromise, or undocumented signing scheme changes in NexaID's protocol.

### No Trust in Nexash's Servers

ZK proof generation happens entirely in the user's browser using WebAssembly. No private inputs are sent to Nexash's servers. The frontend is a static Next.js deployment with no server-side computation involved in proof generation.

### No Trust in HashKey Pay

HSP is used for structured payment receipts alongside ZK-verified on-chain transfers. The ZK payment executes on-chain regardless of HSP's status. If HSP is unavailable, the ZK payment still goes through.

### No Trust in Any Oracle

There is no oracle in Nexash's payment flow. The entire verification chain is:
- NexaID zkTLS (TEE-based, on-chain record)
- Noir ZK circuit (mathematical proof)
- UltraHonk Solidity verifier (deterministic on-chain verification)

No external data feed, price oracle, or KYC service is queried during payment execution.

## What Nexash Verifies

### On-Chain (By Smart Contracts)

- **Identity proof validity** — `IdentityVerifier.verify()` checks the UltraHonk proof cryptographically
- **Policy proof validity** — `PolicyVerifier.verify()` checks the policy proof
- **Policy hash binding** — `policyPubInputs[POL_IDX_POLICY_HASH]` must match `PolicyEngine.getPolicyHash(treasury)`
- **Nullifier freshness** — `usedNullifiers[nullifier]` must be false
- **Proof expiry** — `block.timestamp <= proofTimestamp + PROOF_EXPIRY`
- **Treasury balance** — `IERC20.balanceOf(treasury) >= amount`

### In-Browser (By ZK Circuits)

- **KYC level** — `kycLevel >= min_kyc_level`
- **Jurisdiction membership** — Merkle proof against `allowed_jurisdictions_root`
- **Nullifier correctness** — `nullifier == pedersen_hash(secret, treasury_address)`
- **Attestation existence** — `report_tx_hash != 0`
- **Hash consistency** — `report_tx_hash == report_tx_hash_public`
- **KYC data commitment** — Private inputs match the committed values
- **Amount within limits** — `amount <= spending_limit`
- **Daily limit** — `daily_spent + amount <= daily_spend_limit`

## What Nexash Assumes

### Phala TEE Integrity

Nexash assumes that Phala's TEE infrastructure correctly executes NexaID's attestor code without tampering. If the TEE were compromised, an attacker could create fake attestations. However:
- The attacker would still need to create a real confirmed transaction on HashKey Chain (expensive, traceable)
- The `taskId` binding would need to match (specific to the user's session)
- The compromised attestation would still produce a proof that an attestation tx exists — not that a specific KYC status was verified

In practice, TEE integrity is much harder to compromise than a centralized API.

### Binance KYC Accuracy

Nexash assumes that Binance's KYC API returns accurate information about a user's KYC status. If Binance's KYC system is compromised or inaccurate, Nexash would verify incorrect KYC status. This is a fundamental limitation of any KYC system that relies on third-party verification.

### Noir Circuit Correctness

Nexash assumes that the compiled Noir circuits correctly implement the stated constraints. The circuits are open source and can be audited. The UltraHonk verifier is generated by Barretenberg, which is audited by Aztec Labs.

## Security Properties

| Property | Guarantee |
|---|---|
| **Soundness** | A proof cannot be valid unless all circuit constraints are satisfied |
| **Zero-knowledge** | A valid proof reveals nothing about private inputs beyond what is implied by public inputs |
| **Replay prevention** | Each nullifier can only be used once per treasury |
| **Policy binding** | A proof is only valid for the exact policy it was generated against |
| **Freshness** | Proofs expire after the configured window (default 1 hour) |
| **Non-custodial** | Institution controls treasury keys; Nexash cannot access funds |
